# 数据库备份与恢复

> 数据库跑在 NAS 上、经 cpolar 隧道对外。备份分两条腿：**云端每日自动**（backupDb 云函数 + 定时触发器）与**本地手动**（`scripts/backup-db.js`）。本文是出事时的操作手册。

---

## 一、备份放在哪

| 位置 | 内容 | 保留 | 产生方式 |
|---|---|---|---|
| CloudBase 云存储 `backups/` | 加密压缩包 `<库名>-YYYYMMDD.sql.gz.enc` | 最近 **30 天**，超期自动删 | backupDb 云函数每日 03:00 自动 |
| 本地 `backups/` | 明文 `.sql`（已 gitignore） | 手动清理 | `npm run backup` |

**均不入 Git**。仓库是**公开**的，备份含用户手机号、openid、订单等隐私数据，绝不可提交。

### 为什么要加密

云存储桶权限是「**所有用户可读**」——这是小程序显示头像、故事配图、活动视频所必需的，不能改为私有（一改，用户就看不到别人上传的图了）。因此落到 `backups/` 的文件若是明文，等同于挂在公网上任人下载。

所以备份统一 **gzip + AES-256-GCM 加密**后再上传：

- 密钥由根目录 `.env` 的 `ADMIN_PASSWORD` 经 **scrypt** 派生（每个文件独立随机盐）
- 实现见 `miniprogram/cloudfunctions/backupDb/crypto.js`（云函数与本地脚本共用同一份，避免加解密实现漂移）
- 公网即使拿到文件，也只能看到 `XSBAK` 魔数 + 随机盐 + 密文

> ⚠️ **`ADMIN_PASSWORD` 就是备份的解密钥匙。** 轮换它之前，先把旧备份用旧密码全部解出来落地，否则历史备份永久打不开。请把它同时存一份在密码管理器里。

## 二、日常：什么都不用做

backupDb 云函数每天 03:00（北京时间）自动跑完整流程：导出 → gzip → 加密 → 上传 → 删除 30 天前那份。当前规模约 **17 张表 836 行 / 356 KB / 5 秒**。

### 确认某天有没有备份成功

```bash
npx -y -p @cloudbase/cli tcb storage list backups/ -e cloud1-xingshu-prd-d1cev0fcca864
```

看今天的文件在不在、大小是否正常即可。

### 最大的失效风险：隧道断了

云函数经 cpolar 隧道回连 NAS 上的 MySQL。**隧道一断，备份就静默失败**（云函数报错，但没人看）。历史上已发生过（NAS 的 DHCP 换 IP 导致隧道重连）。建议在 CloudBase 控制台 **云函数 → backupDb → 监控告警**里配一条「执行失败」告警到邮件或短信。

隧道地址若变更：改 `.env` → `npm run sync-db` → 重新部署 backupDb（命令见下）。

## 三、恢复

### 3.1 看有哪些备份

```bash
node scripts/restore-backup.js
```

### 3.2 下载并解密成可用的 .sql

```bash
node scripts/restore-backup.js backups/xingshu_dev-20260722.sql.gz.enc
```

产出落在本地 `backups/` 下的同名 `.sql`。

### 3.3 带校验的恢复（推荐，建议每月演练一次）

```bash
node scripts/restore-backup.js backups/xingshu_dev-20260722.sql.gz.enc --verify
```

`--verify` 会把还原出的 SQL 导入一个临时库 `xingshu_restore_verify_tmp`，统计表数行数后**自动删除临时库**——**不碰**真实库，可以放心跑。

### 3.4 真正写回数据库

⚠️ 备份里含 `DROP TABLE IF EXISTS`，导入会**覆盖目标库的全部现有数据**。执行前请再确认库名。

```bash
mysql -h 33.tcp.cpolar.top -P 11028 -u <user> -p xingshu_dev < backups/xingshu_dev-20260722.sql
```

## 四、手动备份

改版、迁移、批量改数据这类高风险操作前，先手动打一份：

```bash
npm run backup                 # 导出到本地 backups/
node scripts/backup-db.js --verify   # 顺带导入临时库校验
node scripts/upload-backup.js        # 加密上传云存储（默认取最新一份）
```

也可以直接触发云函数（等效于让定时任务提前跑一次）：

```bash
npx -y -p @cloudbase/cli tcb fn invoke backupDb -e cloud1-xingshu-prd-d1cev0fcca864
```

## 五、重新部署 backupDb

改了 `.env`、`crypto.js`、`dump.js` 或 `index.js` 之后：

```bash
npm run sync-db    # 重新生成 db.js / secret.js
npx -y -p @cloudbase/cli tcb fn deploy backupDb -e cloud1-xingshu-prd-d1cev0fcca864 --force
```

定时触发器配置写在 `cloudbaserc.json` 的 `triggers` 字段（cron `0 0 3 * * * *`，7 段式：秒 分 时 日 月 周 年），随部署一起生效。

## 六、备份覆盖不到的东西

| 内容 | 现状 | 说明 |
|---|---|---|
| **云存储文件** | ❌ 未备份 | 头像、故事配图、活动视频。仓库有 `scripts/migrate-storage.js` 可跨环境搬运，但没有定期备份 |
| **`.env` / `.env.prod`** | ❌ 未备份 | 含数据库密码与 `ADMIN_PASSWORD`。丢了则所有云函数 `db.js` 要重配，**且历史备份全部解不开**。请自行存密码管理器 |
| **代码** | ✅ GitHub | `main` + tag + `backup/*` 分支 |
