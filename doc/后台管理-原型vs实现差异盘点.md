## 三、差异分级与实现成本（后续开发参考）

| 档 | 内容 | 新增后端接口 | 风险/备注 |
|---|---|---|---|
| **A 展示对齐**（低风险，多为只读） | 用户列表加互动数/最后活跃/真实姓名列、剩余天数、头像；用户详情加互动统计五格；日记列表加标签列/转发列/标题摘要/"已编辑"；删除确认加互动数警示；作者/ID 搜索 | 用户列表/详情复用已有查询即可补字段；日记标签筛选需 admin 返回标签列表（可复用 `getTags` 逻辑） | 基本纯前端 + 少量只读字段 |
| **B 编辑能力**（中风险，写操作） | 用户详情编辑昵称/真实姓名/手机号；日记编辑标题/正文/标签/权限 | 需新增 `admin.updateUser`、`admin.updateDiary`（含 admin_logs 审计） | 管理员改用户资料/内容，须审计留痕 |
| **C 后台代发日记** 🔒 | 管理员以某用户身份新建日记 | 需新增 `admin.createDiary`（指定 authorId） | 较敏感，代发内容归属他人，谨慎；曾列范围外 |
| **D OpenID/UnionID 展示复制** 🔒 | 用户详情展示并一键复制两 ID + 含义说明 | 无（userDetail 已可返回，需在 SELECT 补 openid/unionid） | 曾列范围外；数据打通用途 |
| **E 分页** | 用户/日记/订单列表真实分页（10/20/50 + 页码） | 各 list handler 加 LIMIT/OFFSET + total | 数据量大后才有必要；当前量小可延后 |

### 建议优先级

1. **A 档**先做——投入小、纯展示对齐，最快缩小"看起来差很多"的观感差距。
2. **B 档**按需——若运营确有"线下核对后改手机号/昵称""下架前改权限"的诉求再做，务必带审计。
3. **C/D/E** 视实际运营需要再评估；C（代发）与 D（ID 复制）此前已被明确排除，无强需求可不做。

> 说明：推荐人展示/修改、他推荐的用户（v2.2）是实现**优于原型**的部分，保留。会员订单历史/开通入口（v2.4/M1.6）已补齐，本盘点不再重复。
