# 小程序前端 — 原型 vs 实现 差异盘点

> 盘点日期：2026-07-04 | 对照物：小程序原型 `project/src/*.jsx`（React，服务于 `http://localhost:4173/`）
> 实现侧：`miniprogram/pages/*` + `miniprogram/components/*`
> 结论先行：**视觉/交互一致度很高**（实现即照原型落地）；真正的差异集中在 **PRD v2.1→v2.3 的功能演进**——实现在原型基础上**做了加法**（微信登录、权限矩阵、退出登录、真实后端），原型缺失的只有 1 个次要快捷入口。

---

## 一、整体关系

原型（`project/`）是 **v1 视觉设计稿**（用 tweak 面板模拟身份，无真实登录/后端）。实现照它落地了暖纸/印章/衬线设计体系与全部页面结构，随后按 PRD v2.1~v2.3 演进：**授权模型换代、权限矩阵收紧、活动模块、退出登录、真实 CloudBase 后端**。因此绝大多数"差异"是**实现领先于原型的有意演进**，而非落地缺失。

原型页面/路由：5 Tab（广场/活动/收藏/我的/会员）+ 路由（list/detail/compose/activity）+ 弹层（filter/poster/purchase/profile/tag-picker/signup/activity-poster）+ ConfirmDialog + Toast。实现一一对应（弹层落为独立组件或页面内 sheet）。

## 三、实现领先于原型的功能（v2.x 演进，非缺失）

1. **微信登录半屏弹窗（login-sheet，v2.3）** —— 替代原型的"微信授权手机号"旧模型；仅取 openid/unionid。
2. **退出登录 + 设置（v2.3）** —— 原型无。
3. **权限矩阵收紧（v2.1）** —— 未登录看列表不看详情（-3 引导登录）；非会员看会员日记 30% 渐隐 + member-guard；原型直接看全文。
4. **带参小程序码真实生成（v2.2）** —— 日记/活动海报扫码直达 + 分享人 ID 绑定推荐人；原型是占位码。
5. **评论二级回复入口（M2.1b）** —— 原型仅展示回复、只能发一级；实现可回复某条评论 + 删除自己的回复。
6. **真实后端** —— CloudBase 云函数 + MySQL；原型是内存 seed 数据。
7. **配图上传** —— compose 支持云存储配图；原型无。

## 四、原型有而实现缺 / 差异的点（真正的 gap）

| 项 | 说明 | 建议 |
|---|---|---|
| 广场右上角「会员」快捷 chip | 原型广场标题右侧有个金色"★会员" chip 直达会员中心；实现广场顶部只有标题+搜索，无此快捷入口 | 次要。底部已有会员 Tab，可加可不加；若要对齐原型，加一个轻量入口即可 |
| 原型「微信授权手机号」文案/流程 | 属 v1 旧模型，已被 v2.3 微信登录取代 | **不是 gap**，勿回退 |
| 原型 detail 评论输入触发仅聚焦、无真实提交 | 原型是静态 mock | 实现已是真实评论/回复，无需对齐 |

## 五、结论

- **视觉与页面结构：高度一致**——实现忠实落地了原型的暖纸/印章/衬线设计体系与全部页面。
- **功能：实现是原型的超集**——沿 PRD v2.1~v2.3 演进出登录、权限矩阵、退出登录、真实后端、活动带参码、评论回复等，原型（v1 稿）自然没有这些。
- **真正的落地缺失只有 1 个次要项**：广场页「会员」快捷 chip（有底部 Tab 兜底，非必需）。
- **一处需避免误判**：原型的"手机号授权"是被 v2.3 有意取代的旧设计，**不应视为待补齐的 gap**。

> 建议：若追求与原型逐像素对齐，仅需补广场「会员」chip 这一个小入口（低优先级）；其余"差异"均为 PRD 演进带来的实现领先，保持现状即可。UI 逐屏观感核对可并入 M2.2 开发者工具人工回归批次。
